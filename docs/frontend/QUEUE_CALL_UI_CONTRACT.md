# Queue Call UI Contract V1

## 1. Purpose

Define the minimum future Store Staff UI contract for calling an existing waiting QueueTicket.

Target flow:

```text
Store staff opens Queue Call page
-> enters queueTicketId
-> optionally enters calledAt, reasonCode, and note
-> UI generates Idempotency-Key
-> UI calls POST /api/v1/stores/{storeId}/queue-tickets/{queueTicketId}/call
-> QueueTicket changes from waiting to called
-> Reservation remains arrived when the ticket is reservation-sourced
-> UI displays the call result
```

This document is contract only. It does not create a Vue page, route, API client, backend API, migration, SQL, seed data, production configuration, runtime fixture data, or production database data.

## 4. Route Contract

Future route:

```text
/stores/:storeId/queue-tickets/call
```

Future page:

```text
QueueCallPage.vue
```

Rationale:

- V1 is a manual QueueTicket call by explicit `queueTicketId`.
- It is not a full Queue Center.
- It does not display a waiting list.
- It does not select the next ticket automatically.
- It does not seat the party.
- It stays inside the Store Staff workflow.

Do not design these routes for V1:

```text
/stores/:storeId/queue
/stores/:storeId/queue/display
/stores/:storeId/queue/call-next
/stores/:storeId/seating/from-queue
```

## 6. Permission Rule

UI visibility rule:

```text
Show Queue Call entry only when actor permissions include queue.call.
Hide Queue Call entry when actor permissions do not include queue.call.
```

Security rule:

```text
Frontend permission checks are display hints only.
Backend Queue Call API plus App Gate remain the final authorization source.
```

The future UI must not assume that hiding a button is sufficient authorization.

Backend gate remains:

```text
@RequireAppGate(appKey = "reservation_queue", permission = "queue.call")
```

`/api/me/apps` V1 remains an app-entry source. The current `permissions` field may be used for the approved Staff Home display rule in this contract, but it must not be treated as a complete capability matrix for unapproved buttons.

## 7. API Call Contract

Endpoint:

```http
POST /api/v1/stores/{storeId}/queue-tickets/{queueTicketId}/call
```

Sources:

| Value | UI/API source |
| --- | --- |
| `storeId` | Route path `/stores/:storeId/...` |
| `queueTicketId` | Required form input, used in API path |
| `Idempotency-Key` | Generated by UI per submit |
| `calledAt` | Optional request body field |
| `reasonCode` | Optional request body field |
| `note` | Optional request body field |

Headers:

```text
Idempotency-Key: queue:call:<uuid>
Content-Type: application/json
Accept: application/json
```

Request body may contain only:

```json
{
  "calledAt": "2030-06-20T03:30:00Z",
  "reasonCode": "TABLE_READY",
  "note": "Call customer near entrance"
}
```

When optional fields are blank, the UI should omit them or send `null` only if the API client normalizes them consistently with existing frontend patterns. The UI must not send empty strings as trusted operational values when a field is intentionally absent.

Forbidden request body fields:

```text
tenantId
storeId
queueTicketId
reservationId
reservationStatus
queueTicketStatus
tableId
tableGroupId
seatingId
cleaningId
turnoverId
skipReason
rejoinReason
status
actorId
actorType
```

## 8. Form Fields

Required:

```text
queueTicketId
```

Optional:

```text
calledAt
reasonCode
note
```

System generated:

```text
Idempotency-Key
```

Form behavior:

- `queueTicketId` is the primary field and should be visually strongest.
- `calledAt` is optional. If blank, the backend application uses current UTC time.
- `calledAt`, when provided, must be exchanged as an ISO8601 instant.
- `reasonCode` is optional operational evidence.
- `note` is optional and visually secondary.
- Submit is disabled while the request is in flight.
- Submit is disabled when `queueTicketId` is blank.
- A new submit attempt uses a new `Idempotency-Key` unless intentionally retrying the same in-flight command.

Forbidden form/body fields:

```text
tenantId
storeId in body
queueTicketId in body
reservationStatus
tableId
tableGroupId
seatingId
cleaningId
turnoverId
skipReason
rejoinReason
status
```

## 9. Success Display Contract

The future UI must display:

```text
queueTicketId
queueTicketNumber
queueTicketStatus
reservationId
reservationCode
reservationStatus
calledAt
holdUntilAt
alreadyCalled
events
idempotency
```

Priority display:

```text
queueTicketNumber
queueTicketStatus = called
holdUntilAt
reservationCode
reservationStatus = arrived
alreadyCalled
```

Fresh success:

```text
queueTicketStatus = called
reservationStatus = arrived
alreadyCalled = false
events contains queue_ticket.called
idempotency.status = completed
idempotency.replayed = false
```

Completed idempotency replay:

```text
queueTicketStatus = called
reservationStatus = arrived
idempotency.status = completed
idempotency.replayed = true
events may be empty
```

The success response is an API DTO projection. The UI must not assume it is a domain object or persistence entity.

## 10. AlreadyCalled Display Contract

Already called is success-like, not failure.

Display:

```text
queueTicketStatus = called
reservationStatus = arrived
alreadyCalled = true
queueTicketNumber
calledAt
holdUntilAt
reservationCode
events
idempotency.status = completed
```

Rules:

- `alreadyCalled=true` must not render a failure panel.
- `events` may be empty.
- The result should still highlight `queueTicketStatus=called` and the existing `queueTicketNumber`.
- The UI should show `calledAt` and `holdUntilAt`.
- The UI may show `idempotency.replayed` when present.
- The UI must not create or change QueueTicket call evidence locally.

## 12. Mobile-first Rules

The future Queue Call page must be mobile-first:

- Single-column layout.
- `queueTicketId` field appears first and is visually prominent.
- `calledAt` is optional and secondary to `queueTicketId`.
- `reasonCode` and `note` are visually secondary.
- Submit button is clear and thumb-friendly.
- Loading state prevents duplicate accidental taps.
- Disabled state is obvious when required values are missing or the request is in flight.
- Success result highlights `queueTicketNumber`.
- Success result shows `queueTicketStatus=called`.
- Success result shows `holdUntilAt`.
- Success result shows `reservationStatus=arrived` when present.
- `alreadyCalled=true` is visible and treated as success-like.
- Error panel clearly shows `error.code` and `error.messageKey`.
- Text must fit within cards, buttons, and result panels on mobile.
- Do not add complex back-office page layout.
- Do not add Queue list, Queue workbench, Queue skip, Queue rejoin, display screen, Seating from Queue, table map, drag-and-drop, No-show, or Cancellation UI.
