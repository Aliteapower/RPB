## 4. API Client

- Endpoint: `POST /api/v1/stores/{storeId}/queue-tickets/{queueTicketId}/call`
- Idempotency-Key: generated by the page and sent as `Idempotency-Key`
- Body allowlist: `calledAt`, `reasonCode`, `note`
- Blank optional fields are omitted by the API client.
- `storeId` and `queueTicketId` are used only in the path, not in the body.
- Errors preserve `error.code` and `error.messageKey`; unknown or invalid responses fall back to `UNKNOWN_ERROR / queue.call.unknown_error`.

## 7. calledAt Behavior

- `calledAt` may be left blank.
- Blank `calledAt` is omitted from the request body, so the backend application resolves current UTC time.
- Filled `calledAt` uses `datetime-local` input and is converted with `toISOString()` before submission.
- Invalid `calledAt` produces a local `INVALID_COMMAND / queue.call.called_at_invalid` error display.

## 11. AlreadyCalled Display

- `alreadyCalled=true` renders inside the success panel.
- It is not displayed as a failure.
- Empty `events` display as `[]`.
- Existing `calledAt`, `holdUntilAt`, queue number, and idempotency fields remain visible.

## 12. Error Display

- Error panel displays `错误代码：{{ apiError.error.code }}`.
- Error panel displays `消息键：{{ apiError.error.messageKey }}`.
- The UI does not replace backend `messageKey` with hardcoded business copy.
- Network or invalid response fallback is `UNKNOWN_ERROR / queue.call.unknown_error`.

## 13. Mobile-first Handling

- Single-column shell.
- `排队票 ID` field is first and visually prominent.
- `叫号时间` is optional and secondary.
- `原因代码 / 备注` are inside a secondary details block.
- Submit button is thumb-friendly and disabled during loading or when the required ID is blank.
- Success result emphasizes `排队号码`, `called`, and `holdUntilAt`.
- Error result wraps long message keys safely.
- No complex queue workbench, display screen, table map, drag-and-drop, or desktop-only layout was introduced.
