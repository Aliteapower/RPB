# Reservation CheckIn UI Contract V1

## 1. Purpose

Define the minimum future Store Staff UI contract for Reservation CheckIn.

Target flow:

```text
Store staff opens CheckIn page
-> enters reservationId
-> optionally enters arrivedAt, reasonCode, note
-> submits CheckIn
-> UI calls POST /api/v1/stores/{storeId}/reservations/{reservationId}/check-in
-> Reservation changes confirmed -> arrived
-> UI displays status=arrived, alreadyArrived, arrivedAt, and idempotency
```

This document is contract only. It does not create a Vue page, route, API client, backend API, migration, seed data, or production configuration.

## 2. Scope

In scope for the future minimum UI:

- Manual `reservationId` input.
- Optional `arrivedAt` input.
- Optional `reasonCode` input.
- Optional `note` input.
- Automatic `Idempotency-Key` generation.
- CheckIn submit.
- Success display.
- Already-arrived display.
- Error display.
- Staff Home entry rule for a later UI implementation.
- Mobile-first layout rules.
- Future implementation test contract.

Out of scope:

## 4. Route Contract

Future route:

```text
/stores/:storeId/reservations/check-in
```

Future page:

```text
ReservationCheckInPage.vue
```

Rationale:

- V1 has no Reservation list, calendar, or search.
- Store staff can manually enter `reservationId`.
- The page stays inside the existing Store Staff workflow.
- The route does not introduce a separate `check-ins` resource.

Do not design these routes for V1:

```text
/stores/:storeId/check-ins
/stores/:storeId/queue
/stores/:storeId/reservations/calendar
```

## 5. Staff Home Entry Rule

This contract does not modify Staff Home.

In a future UI implementation, Staff Home may add:

```text
Check In Reservation
```

Route target:

```text
/stores/:storeId/reservations/check-in
```

Display condition:

```text
reservation_queue app is visible and enabled for the store
and
reservation_queue permissions contains reservation.check_in
```

Do not show the CheckIn entry when:

- `/api/me/apps` does not return `reservation_queue`.
- `reservation_queue.entryVisible` is false.
- `reservation_queue.permissions` does not contain `reservation.check_in`.

Staff Home must still not add Queue, Seating, No-show, Cancellation, Reservation list, or Reservation calendar entries in the CheckIn UI slice.

## 6. Permission Rule

UI visibility rule:

```text
Show CheckIn UI entry only when actor permissions include reservation.check_in.
Hide CheckIn UI entry when actor permissions do not include reservation.check_in.
```

Security rule:

```text
Frontend permission checks are display hints only.
Backend CheckIn API + App Gate remain the final authorization source.
```

The future UI must not assume that hiding a button is sufficient authorization.

Backend gate remains:

```text
@RequireAppGate(appKey = "reservation_queue", permission = "reservation.check_in")
```

## 7. API Call Contract

Endpoint:

```http
POST /api/v1/stores/{storeId}/reservations/{reservationId}/check-in
```

Sources:

| Value | UI/API source |
| --- | --- |
| `storeId` | Route path `/stores/:storeId/...` |
| `reservationId` | Required form input, used in API path |
| `Idempotency-Key` | Generated by UI per submit |
| `arrivedAt` | Optional request body field |
| `reasonCode` | Optional request body field |
| `note` | Optional request body field |

Headers:

```text
Idempotency-Key: reservation:check-in:<uuid>
Content-Type: application/json
Accept: application/json
```

Request body may contain only:

```json
{
  "arrivedAt": "2026-06-21T04:00:00Z",
  "reasonCode": "customer_arrived",
  "note": "Guest is at host stand"
}
```

When optional fields are blank, the UI may omit them or send null only if the API client normalizes them consistently with existing frontend patterns. The UI must not send empty strings as trusted operational values when a field is intentionally absent.

Forbidden request body fields:

```text
tenantId
storeId
reservationId
actorId
actorType
queueTicketId
seatingId
tableId
tableGroupId
noShowAt
cancelledAt
status
```

## 8. Form Fields

Required:

```text
reservationId
```

Optional:

```text
arrivedAt
reasonCode
note
```

System generated:

```text
Idempotency-Key
```

Form behavior:

- `reservationId` is the primary field and should be visually strongest.
- `arrivedAt` may be left blank so the backend application clock resolves arrival time.
- `reasonCode` is optional operational evidence.
- `note` is optional and visually secondary.
- Submit is disabled while the request is in flight.
- A new submit attempt uses a new `Idempotency-Key` unless intentionally retrying the same in-flight command.

Forbidden form fields:

```text
tenantId
storeId in body
queueTicketId
seatingId
tableId
tableGroupId
noShowAt
cancelledAt
status
```

## 9. Success Display

The future UI must display:

```text
reservationId
reservationCode
status
arrivedAt
alreadyArrived
events
idempotency.status
idempotency.replayed
```

Priority display:

```text
status = arrived
reservationCode
alreadyArrived
```

Fresh success:

```text
status = arrived
alreadyArrived = false
events contains reservation.arrived
idempotency.status = completed
idempotency.replayed = false
```

Already-arrived success-like response:

```text
status = arrived
alreadyArrived = true
events may be empty
idempotency.status = completed
```

Already-arrived must not be displayed as a failure.

Completed idempotency replay:

```text
status = arrived
idempotency.status = completed
idempotency.replayed = true
```

## 10. Error Display

The future UI must display:

```text
error.code
error.messageKey
```

The UI must not replace `messageKey` with hardcoded business copy.

Minimum error cases to support:

```text
MISSING_IDEMPOTENCY_KEY
RESERVATION_NOT_FOUND
RESERVATION_STATUS_NOT_CONFIRMED
RESERVATION_CANNOT_CHECK_IN_CANCELLED
RESERVATION_CANNOT_CHECK_IN_NO_SHOW
RESERVATION_CANNOT_CHECK_IN_COMPLETED
RESERVATION_CANNOT_CHECK_IN_SEATED
IDEMPOTENCY_CONFLICT
IDEMPOTENCY_IN_PROGRESS
IDEMPOTENCY_FAILED_REQUIRES_NEW_KEY
FORBIDDEN
STORE_SCOPE_MISMATCH
APP_DISABLED
TENANT_APP_NOT_ENABLED
TENANT_APP_EXPIRED
STORE_APP_NOT_ENABLED
PERMISSION_DENIED
UNKNOWN_ERROR
```

Compatibility alias:

```text
RESERVATION_INVALID_STATUS
```

If a UI test or local helper uses `RESERVATION_INVALID_STATUS`, it must map to the same display behavior as `RESERVATION_STATUS_NOT_CONFIRMED` and still show the returned `error.code` and `error.messageKey`.

## 11. Mobile-first Rules

The future CheckIn page must be mobile-first:

- Single-column layout.
- `reservationId` field appears first and is visually prominent.
- `arrivedAt` is optional and may be collapsed or secondary.
- `reasonCode` and `note` are secondary.
- Submit button is clear and thumb-friendly.
- Loading state prevents duplicate accidental taps.
- Success result highlights `status=arrived`.
- `reservationCode` is easy to read and copy.
- `alreadyArrived=true` is visible and treated as success-like.
- Error panel clearly shows `error.code` and `error.messageKey`.
- Text must fit within cards/buttons on mobile.
- Do not add complex dashboard, table map, calendar, or list UI.

## 12. Future Implementation Test Contract

Route tests:

- `/stores/:storeId/reservations/check-in` opens the future page.
- Staff Home link points to the same store route when the entry is permitted.

Form tests:

- `reservationId` is required.
- `arrivedAt` may be blank.
- `reasonCode` may be blank.
- `note` may be blank.
- `Idempotency-Key` is generated automatically.
- Request body excludes forbidden fields.

Permission tests:

- Staff Home shows CheckIn entry when `reservation_queue` is visible and `permissions` contains `reservation.check_in`.
- Staff Home hides CheckIn entry when `permissions` does not contain `reservation.check_in`.
- UI permission logic is display-only; backend App Gate denial still renders returned API error.

API tests:

- Calls `POST /api/v1/stores/{storeId}/reservations/{reservationId}/check-in`.
- Sends `reservationId` only in the path.
- Sends `storeId` only in the path.
- Sends `Idempotency-Key`.
- Sends only `arrivedAt`, `reasonCode`, and `note` in the body.

Success tests:

- Confirmed reservation check-in displays `status=arrived`.
- Displays `reservationCode`.
- Displays `arrivedAt`.
- Displays `alreadyArrived=false`.
- Displays `events`.
- Displays `idempotency.status=completed`.

Already-arrived tests:

- Displays `alreadyArrived=true`.
- Treats the response as success-like.
- Does not show a failure panel.

Error tests:

- Displays `error.code`.
- Displays `error.messageKey`.
- Does not hardcode translated business copy instead of `messageKey`.

Boundary tests:

- No Queue UI.
- No Seating UI.
- No No-show UI.
- No Cancellation UI.
- No Reservation list/calendar.
- No table assignment UI.
- No backend API changes.
- No migration changes.
