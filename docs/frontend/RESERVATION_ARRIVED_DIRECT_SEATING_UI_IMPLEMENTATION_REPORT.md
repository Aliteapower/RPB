## 4. API Client

Created:

```text
src/api/reservationArrivedDirectSeatingApi.ts
```

Function:

```text
seatArrivedReservation(storeId, reservationId, request, idempotencyKey, fetcher)
```

Endpoint:

```http
POST /api/v1/stores/{storeId}/reservations/{reservationId}/seating/direct
```

Idempotency-Key:

```text
Sent in Idempotency-Key request header.
Generated by the page before submit.
```

Body allowlist:

```text
tableId
tableGroupId
overrideReasonCode
overrideNote
note
```

The client `toApiBody` returns only the allowlist fields and does not include `tenantId`, body `storeId`, body `reservationId`, `queueTicketId`, `walkInId`, `checkInAt`, `noShowAt`, `cancelledAt`, `cleaningId`, `turnoverId`, or business `status`.

Error behavior:

- Backend error envelopes are surfaced through `ReservationArrivedDirectSeatingApiError`.
- The page displays returned `error.code` and `error.messageKey`.
- Invalid, network, or unclassified local fallback uses `UNKNOWN_ERROR`.

Implemented frontend validation:

- `reservationId` required.
- `tableId` / `tableGroupId` exactly one required.
- Both empty returns local `RESOURCE_SELECTION_REQUIRED`.

- Submit button is disabled when resource selection is missing or conflicting.
- Backend remains final validation source.

## 13. Mobile-first Handling

Implemented with the existing Store Staff UI pattern:

- Single-column `page-shell`.
- `reservationId` field appears first and is visually prominent.
- `tableId` / `tableGroupId` selection is grouped and clearly separate.

- Success panel highlights `reservationStatus=seated`.

- Text wraps inside result panels.
