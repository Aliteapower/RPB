# Reservation Arrived Direct Seating UI Contract V1

## 1. Purpose

Define the minimum future Store Staff UI contract for seating an arrived Reservation directly.

Target flow:

```text
Store staff opens Direct Seating page
-> enters reservationId
-> enters exactly one resource: tableId or tableGroupId
-> optionally enters overrideReasonCode, overrideNote, and note
-> UI generates Idempotency-Key
-> UI calls POST /api/v1/stores/{storeId}/reservations/{reservationId}/seating/direct
-> Reservation changes arrived -> seated
-> Seating and SeatingResource are created
-> table or table group member tables become occupied
-> UI displays seated result
```

This document is contract only. It does not create a Vue page, route, API client, backend API, migration, SQL, seed data, production configuration, or runtime fixture data.

## 2. Scope

In scope for the future minimum UI:

- Manual `reservationId` input.
- Manual `tableId` input.
- Manual `tableGroupId` input.
- Exactly-one resource selection validation between `tableId` and `tableGroupId`.
- Optional `overrideReasonCode` input.
- Optional `overrideNote` input.
- Optional `note` input.

## 4. Route Contract

Future route:

```text
/stores/:storeId/reservations/seating/direct
```

Future page:

```text
ReservationArrivedDirectSeatingPage.vue
```

Rationale:

- V1 has no Reservation list, calendar, search, or table map.
- Store staff can manually enter `reservationId`.
- Store staff can manually enter `tableId` or `tableGroupId`.
- The route stays inside the Store Staff workflow.
- The route does not create Queue, calendar, table-map, or auto-seating entry points.

Do not design these routes for V1:

```text
/stores/:storeId/queue
/stores/:storeId/reservations/calendar
/stores/:storeId/table-map
/stores/:storeId/seating/auto
```

## 5. Staff Home Entry Rule

This contract does not modify Staff Home.

In a future UI implementation, Staff Home may add:

```text
Seat Arrived Reservation
```

Route target:

```text
/stores/:storeId/reservations/seating/direct
```

Display condition:

```text
reservation_queue app is visible and enabled for the store
and
reservation_queue permissions contains reservation.seat
```

Do not show the entry when:

- `/api/me/apps` does not return `reservation_queue`.
- `reservation_queue.entryVisible` is false.
- `reservation_queue.permissions` does not contain `reservation.seat`.

Staff Home must still not add Queue, Reservation list, Reservation calendar, table map, auto assignment, No-show, Cancellation, Cleaning, or Turnover entries in this UI slice.

## 6. Permission Rule

UI visibility rule:

```text
Show the Direct Seating entry only when actor permissions include reservation.seat.
Hide the Direct Seating entry when actor permissions do not include reservation.seat.
```

Security rule:

```text
Frontend permission checks are display hints only.
Backend Direct Seating API plus App Gate remain the final authorization source.
```

The future UI must not assume that hiding a button is sufficient authorization.

Backend gate remains:

```text
@RequireAppGate(appKey = "reservation_queue", permission = "reservation.seat")
```

## 7. API Call Contract

Endpoint:

```http
POST /api/v1/stores/{storeId}/reservations/{reservationId}/seating/direct
```

Sources:

| Value | UI/API source |
| --- | --- |
| `storeId` | Route path `/stores/:storeId/...` |
| `reservationId` | Required form input, used in API path |
| `Idempotency-Key` | Generated by UI per submit |
| `tableId` | Optional request body field, mutually exclusive with `tableGroupId` |
| `tableGroupId` | Optional request body field, mutually exclusive with `tableId` |
| `overrideReasonCode` | Optional request body field |
| `overrideNote` | Optional request body field |
| `note` | Optional request body field |

Headers:

```text
Idempotency-Key: <generated-key>
Content-Type: application/json
Accept: application/json
```

Request body may contain only:

```json
{
  "tableId": "60000000-0000-0000-0000-000000000951",
  "overrideReasonCode": "manager_override",
  "overrideNote": "Approved by shift lead",
  "note": "Seat near window"
}
```

Or:

```json
{
  "tableGroupId": "70000000-0000-0000-0000-000000000951",
  "overrideReasonCode": "manager_override",
  "overrideNote": "Approved by shift lead",
  "note": "Two joined tables"
}
```

The UI must omit blank optional fields or normalize them consistently with existing frontend API patterns. The UI must not send empty strings as trusted operational values when a field is intentionally absent.

Forbidden request body fields:

```text
tenantId
storeId
reservationId
queueTicketId
walkInId
checkInAt
noShowAt
cancelledAt
cleaningId
turnoverId
status
```

## 8. Form Fields

Required:

```text
reservationId
```

Resource selection:

```text
tableId optional
tableGroupId optional
```

Resource rule:

```text
tableId / tableGroupId exactly one required
```

Optional:

```text
overrideReasonCode
overrideNote
note
```

System generated:

```text
Idempotency-Key
```

Form behavior:

- `reservationId` is the primary field and should be visually strongest.
- Resource selection must clearly present `tableId` and `tableGroupId` as mutually exclusive choices.
- Submit is blocked when both `tableId` and `tableGroupId` are blank.
- Submit is blocked when both `tableId` and `tableGroupId` are filled.
- Optional fields are visually secondary.
- Submit is disabled while the request is in flight.
- A new submit attempt uses a new `Idempotency-Key` unless intentionally retrying the same in-flight command.

Forbidden form/body fields:

```text
tenantId
storeId in body
reservationId in body
queueTicketId
walkInId
checkInAt
noShowAt
cancelledAt
cleaningId
turnoverId
status
```

## 9. Resource Selection Rule

The future UI must enforce the same minimum resource selection rule as the API:

```text
Exactly one of tableId or tableGroupId is required.
```

Allowed examples:

```json
{ "tableId": "60000000-0000-0000-0000-000000000951" }
```

```json
{ "tableGroupId": "70000000-0000-0000-0000-000000000951" }
```

Disallowed examples:

```json
{ "tableId": null, "tableGroupId": null }
```

```json
{
  "tableId": "60000000-0000-0000-0000-000000000951",
  "tableGroupId": "70000000-0000-0000-0000-000000000951"
}
```

The UI should prevent the invalid cases before submit. Backend validation remains final and may return `RESOURCE_SELECTION_REQUIRED` or `RESOURCE_SELECTION_CONFLICT`.

## 10. Success Display Contract

The future UI must display:

```text
reservationId
reservationCode
reservationStatus
seatingId
seatingStatus
resourceType
resourceId
alreadySeated
events
idempotency
```

Priority display:

```text
reservationStatus = seated
reservationCode
seatingId
resourceType / resourceId
alreadySeated
```

Fresh success:

```text
reservationStatus = seated
alreadySeated = false
events contains reservation.seated
events contains seating.created
events contains table.occupied
idempotency.status = completed
idempotency.replayed = false
```

Completed idempotency replay:

```text
reservationStatus = seated
idempotency.status = completed
idempotency.replayed = true
```

The success response is an API DTO projection. The UI must not assume it is a domain object or persistence entity.

## 11. Already Seated Display Contract

Already-seated is success-like, not failure.

Display:

```text
reservationStatus = seated
alreadySeated = true
reservationCode
seatingId
resourceType
resourceId
events
idempotency.status = completed
```

Rules:

- `alreadySeated=true` must not render a failure panel.
- `events` may be empty.
- The result should still highlight `reservationStatus=seated`.
- The UI may show `idempotency.replayed` when present.

If the backend returns `RESERVATION_SEATED_WITHOUT_ACTIVE_SEATING`, that is an error response for an inconsistent seated state without active seating and must be displayed as `error.code` plus `error.messageKey`.

## 12. Error Display Contract

The UI must not replace `messageKey` with hardcoded business copy.

The UI should display the actual backend `error.code`. Compatibility aliases may be used only in tests, fixtures, or local display grouping; they must not hide or rewrite the returned backend code.

## 13. Mobile-first Rules

The future Direct Seating page must be mobile-first:

- Single-column layout.
- `reservationId` input appears first and is visually prominent.
- Resource selection clearly shows `tableId` and `tableGroupId` as exactly-one inputs.
- Optional fields are collapsed, grouped, or visually secondary.
- Submit button is clear and thumb-friendly.
- Loading state prevents duplicate accidental taps.
- Disabled state is obvious when required values are missing or the request is in flight.
- Success result highlights `reservationStatus=seated`.
- `reservationCode`, `seatingId`, `resourceType`, and `resourceId` are easy to read.
- `alreadySeated=true` is visible and treated as success-like.
- Error panel clearly shows `error.code` and `error.messageKey`.
- Text must fit within cards, buttons, and result panels on mobile.
- Do not add complex back-office page layout.
- Do not add calendar, list, table map, drag-and-drop, Queue, or auto-assignment UI.
