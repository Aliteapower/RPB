## 3. Rules / Policies Implemented

- `ReservationArrivedToQueueRule` accepts only `ReservationStatus.ARRIVED` for fresh queue creation.
- Terminal and incompatible reservation statuses map to explicit application errors.
- `QueueGroupSelectionRule` validates an active QueueGroup and party-size coverage.
- `QueueTicketNumberPolicy` generates backend-owned integer ticket numbers and queue positions from active waiting tickets.
- Ticket number overflow is mapped to `QUEUE_TICKET_NUMBER_CONFLICT`.

## 4. Reservation Status Behavior

- Reservation status stays `arrived`.
- No new reservation state was added.
- No reservation status transition to queued was introduced.
- Queued state is expressed by `QueueTicket.status = waiting`.

## 5. QueueTicket Source / Association

- `QueueTicket` now supports reservation association through the existing `reservation_id` concept.
- Reservation-sourced tickets set `reservationId`.
- Walk-in source remains supported through `walkInId`.
- Source validation keeps reservation and walk-in mutually exclusive.
- No `source_type` column or schema change was introduced.

## 6. QueueGroup Behavior

- QueueGroup can be selected by explicit party-size group code or by party size.
- QueueGroup must be active.
- QueueGroup must cover the Reservation party size.
- Missing or mismatched QueueGroup returns explicit application errors.

## 7. Ticket Number Behavior

- Ticket number is generated by backend policy.
- Caller-provided ticket numbers are not accepted.
- Next ticket number is based on active waiting queue tickets.
- Queue position is generated as the next active waiting position.
- Ticket number remains an integer.

## 8. AlreadyQueued Behavior

- V1 active queued status is `waiting`.
- If an active waiting QueueTicket already exists for the reservation:
  - The service returns success-like `alreadyQueued = true`.
  - No duplicate QueueTicket is saved.
  - No duplicate `reservation.queued` event is written.
  - No duplicate `queue_ticket.created` event is written.
  - No duplicate transition log or audit log is written for the business action.

## 9. Idempotency Behavior

- Action key is `queue_arrived_reservation`.
- Request hash includes store, reservation, actor, optional QueueGroup code, note, and source.
- Existing in-progress commands return retry-later behavior.
- Hash conflicts return idempotency conflict.
- Completed idempotency records replay from the stored snapshot.
- Already-queued idempotency completion stores the existing QueueTicket snapshot.

## 10. Events / Transition / Audit

- Success writes `reservation.queued`.
- Success writes `queue_ticket.created`.
- Success writes a QueueTicket state transition from `none` to `waiting`.
- Success audit operation is `reservation.queue`.
- Failure audit operation is `reservation.queue.failed`.
- Failure audit is best-effort and does not mask the original application error.

## 11. Success Cases

- Arrived Reservation can be queued successfully.
- Explicit matching QueueGroup code is accepted.
- QueueGroup can be inferred from party size.
- Ticket number and queue position are generated by backend.
- Already-queued waiting reservation returns success-like result without duplicate writes.
- Completed idempotency record replays the original result.

## 12. Failure Cases

- Missing or invalid command.
- Missing idempotency key.
- Store not found.
- Store scope mismatch.
- Store access denied.
- Reservation not found.
- Reservation scope mismatch.
- Reservation status not arrived.
- Reservation is seated, cancelled, no-show, or completed.
- QueueGroup not found.
- QueueGroup inactive.
- QueueGroup party-size mismatch.
- QueueTicket number conflict.
- Business event, transition log, audit log, or persistence write failure.
- Idempotency conflict or command in progress.
