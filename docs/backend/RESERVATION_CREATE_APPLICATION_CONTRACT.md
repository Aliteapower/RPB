# Reservation Create Application Contract V1

## 1. Selected Slice

Selected vertical slice:

```text
Create Reservation
```

Business flow:

```text
Store staff creates reservation request
-> validate StoreScope
-> resolve or create Customer identity
-> validate party size
-> validate reservation time range
-> check Store policy
-> check duplicate reservation rule
-> check reservation capacity availability
-> create Reservation
-> set status confirmed
-> write BusinessEvent
-> write StateTransitionLog
-> write AuditLog
-> complete Idempotency
-> return application result
```

Create Reservation V1 creates a future capacity intent. It does not check in the customer, create a queue ticket, assign a table, create seating, or start no-show handling.

## 2. Scope

In scope:

- Application contract for Store staff creating a Reservation.
- Store-scoped capacity intent: Store + business date + time range + party size.
- Tenant-scoped Customer identity resolution or temporary Customer creation boundary.
- Reservation status set to `confirmed`.
- BusinessEvent, StateTransitionLog, AuditLog, and Idempotency boundaries.
- Store policy usage for reservation hold and expected dining duration.
- UTC storage and ISO8601 command input boundary.
- Repository Port and Rule / Policy / Validator requirements.
- Test contract for a later implementation round.

Create Reservation result should include at least:

- `reservationId`
- `reservationCode`
- `customerId`
- `partySize`
- `reservedStartAt`
- `reservedEndAt`
- `holdUntilAt`
- `businessDate`
- `status = confirmed`
- event codes
- idempotency status and replay flag

## 3. Non-Scope

Out of scope:

- Reservation CheckIn.
- Reservation Seating.
- Reservation Queue fallback.
- Reservation No-show.
- Reservation Cancel.
- Reservation Update.
- Reservation List.
- Reservation Calendar.
- Table assignment.
- TableLock creation.
- ReservationPreassignment workflow.
- QueueTicket creation.
- Seating creation.
- API implementation.
- Controller.
- API DTO.
- Vue UI.

Explicit boundaries:

- Create Reservation must not lock a specific DiningTable or TableGroup by default.
- Create Reservation may leave `reservation_preassignments` as a future extension boundary, but this slice does not write preassignment rows.

## 4. Command Design

Command:

```text
CreateReservationCommand
```

Required fields:

| Field | Source | Required | Notes |
| --- | --- | --- | --- |
| `tenantId` | server context | Yes | Trusted Tenant scope from actor/security context. |
| `storeId` | path/server context | Yes | Store scope must belong to Tenant scope. |
| `partySize` | request/application input | Yes | Must be positive. |
| `reservedStartAt` | request/application input | Yes | ISO8601 instant, normalized to UTC. |
| `reservedEndAt` | request/application input | Yes | ISO8601 instant, normalized to UTC. |
| `idempotencyKey` | request header/application input | Yes | Required for create. |
| `actorId` | server context | Yes | Staff/system actor id. |
| `actorType` | server context | Yes | Expected values align with `staff`, `customer`, `integration`, `system` source boundary. |

Optional fields:

| Field | Notes |
| --- | --- |
| `customerId` | Existing Tenant-scoped customer. |
| `customerName` | Used when resolving or creating a temporary/no-phone Customer. |
| `customerNickname` | Optional lookup aid for no-phone Customer. |
| `phoneE164` | Nullable; must be E.164 when present. |
| `note` | Staff/customer note. |
| `reservationCode` | Optional external or precomputed code; otherwise generated by ReservationCodePolicy. |
| `source` | Defaults to `staff` for store staff create. |
| `reasonCode` | Optional future operational reason boundary; not required for normal create. |

Forbidden fields:

- `queueTicketId`
- `seatingId`
- `tableId`
- `tableGroupId`
- `checkInAt`
- `noShowAt`

Reason:

Create Reservation V1 does not create Queue, does not seat the guest, does not check in the guest, and does not preassign a concrete resource.

## 5. Application Service Boundary

Application Service:

```text
ReservationCreateApplicationService
```

Responsibilities:

1. Validate command presence and structural requirements.
2. Build `TenantScope` and `StoreScope`.
3. Check idempotency using action `create_reservation`.
4. Validate Store access and Store operational status.
5. Load Store policy.
6. Resolve Customer identity:
   - existing Customer by `customerId`
   - existing Customer by `phoneE164`
   - create temporary/no-phone Customer from name/nickname/note when needed
7. Validate phone using nullable E.164 rule.
8. Validate `partySize`.
9. Validate reservation time range.
10. Derive Store-local `businessDate` from Store timezone.
11. Apply `ReservationHoldPolicy` to compute `holdUntilAt`.
12. Apply expected dining duration policy if `reservedEndAt` is derived by caller in later API layer; in this contract, command already carries both start and end.
13. Check duplicate active reservation for same Store + Customer + time range.
14. Check reservation capacity availability for Store + time range + party size.
15. Generate or validate `reservationCode`.
16. Create Reservation.
17. Set Reservation status to `confirmed`.
18. Write BusinessEvent.
19. Write StateTransitionLog.
20. Write AuditLog.
21. Complete idempotency with response snapshot.
22. Return application result.

Not responsible for:

- API parsing.
- HTTP status mapping.
- UI message text.
- SQL implementation.
- Repository implementation.
- CheckIn.
- Seating.
- Queue.
- No-show.
- Cancellation.
- Table assignment.
- Payment.
- POS.
- Marketing.
- Full Customer search API.
- Store switching.

## 6. Required Repository Ports

Only minimum ports for this slice are required. Do not create mechanical CRUD contracts.

### StoreRepositoryPort

Required method:

```text
findById(StoreScope)
```

Purpose:

- Verify Store exists.
- Verify Store belongs to Tenant scope.
- Load Store timezone, locale, date format, time format, currency, and operational status.

### StorePolicyRepositoryPort

Required method:

```text
findByStore(StoreScope)
```

Purpose:

- Load current Store policy.
- Use `reservation_hold_minutes`.
- Use `expected_dining_minutes` where needed.
- Use Store policy values needed by capacity checks.

### CustomerRepositoryPort

Required methods:

```text
findById(TenantScope, customerId)
findByPhone(TenantScope, phoneE164)
save(TenantScope, customer)
```

Requirements:

- Must support nullable phone.
- Must keep Customer uniqueness inside Tenant.
- Must not require Store scope for Customer identity.
- Must not expose Member, loyalty, marketing, or payment concepts.

### ReservationRepositoryPort

Required methods:

```text
save(StoreScope, Reservation)
existsDuplicateActiveReservation(StoreScope, customerId, TimeRange)
findActiveCapacityUsage(StoreScope, TimeRange)
nextReservationCode(StoreScope)
```

Purpose:

- Persist the Reservation aggregate.
- Check same-customer active duplicate reservation rule.
- Read active capacity usage for the target time range.
- Generate Store-scoped reservation code if command did not provide one.

Notes:

- Active duplicate statuses are `confirmed`, `arrived`, and `seated`.
- Capacity usage must include active Reservations that hold future capacity.
- Full arbitrary overlap is a Rule/Repository query concern, not only the V001 identical-slot partial unique index.

### BusinessEventRepositoryPort

Required method:

```text
append(StoreScope, BusinessEvent)
```

Purpose:

- Append `reservation.created` and/or `reservation.confirmed`.

### StateTransitionLogRepositoryPort

Required method:

```text
append(StoreScope, StateTransitionLog)
```

Purpose:

- Record Reservation transition `none -> confirmed` or internal `draft -> confirmed`.

### AuditLogRepositoryPort

Required method:

```text
append(StoreScope, AuditLog)
```

Purpose:

- Record `reservation.create`.
- Record critical failure audit where the contract requires it.

### IdempotencyRepositoryPort

Required methods:

```text
findOrStart(scope, source, action, key, requestHash, expiresAt)
complete(record, targetRef, responseSnapshot)
fail(record, failureReason)
```

Action:

```text
create_reservation
```

Purpose:

- Deduplicate retried Create Reservation commands.
- Replay completed same-hash result.
- Reject same-key different-hash conflicts.

## 7. Required Rules / Policies / Validators

Required:

| Component | Type | Purpose |
| --- | --- | --- |
| `StoreAccessPolicy` | Policy | Actor must have Tenant and Store scope access for Reservation create. |
| `CustomerIdentityRule` | Rule | Existing or temporary/no-phone Customer identity must be sufficient for Reservation. |
| `CustomerPhoneRule` | Rule | Phone is nullable; if present it must be E.164 and Tenant-scoped unique where applicable. |
| `ReservationAvailabilityRule` | Rule | Store has capacity for requested time range and party size. |
| `ReservationDuplicateRule` | Rule | Same Tenant + Store + Customer + overlapping active slot is blocked. |
| `ReservationHoldPolicy` | Policy | Compute hold window; Store configurable, V1 default 15 minutes after reserved start. |
| `ReservationTimeRangeRule` | Rule | `reservedEndAt > reservedStartAt`; start is not in the past; instants are valid. |
| `ReservationCodePolicy` | Policy | Generate or validate Store-scoped `reservationCode`. |
| `StoreTimeZoneRule` | Rule | Resolve Store timezone and business date; storage remains UTC. |
| `AuditRule` | Rule | Decide success/failure audit payload. |
| `BusinessEventRule` | Rule | Validate required Reservation event codes. |
| `StateTransitionRule` | Rule | Validate `none/draft -> confirmed`. |
| `IdempotencyRule` | Rule | Apply create replay/conflict/failure behavior. |

Forbidden in this slice:

- `QueueCallingRule`
- `QueueRejoinRule`
- `TableAssignmentRule`
- `SeatingSourceValidator`
- `CleaningResourceValidator`
- `NoShowPolicy`

## 8. State Boundary

Reservation status after successful Create Reservation:

```text
confirmed
```

State transition:

```text
none -> confirmed
```

Alternative internal implementation:

```text
none -> draft -> confirmed
```

If an internal draft is used, the application result and externally observable state for this slice must still be:

```text
confirmed
```

This slice must not expose or implement a draft workflow.

Illegal state behavior in this slice:

- Create Reservation must not produce `arrived`.
- Create Reservation must not produce `seated`.
- Create Reservation must not produce `cancelled`.
- Create Reservation must not produce `no_show`.
- Create Reservation must not produce `completed`.

## 9. Event Boundary

Required event options:

Preferred V1 event set:

```text
reservation.created
reservation.confirmed
```

Rationale:

- `reservation.created` records the business object creation.
- `reservation.confirmed` records the capacity-holding state.
- This supports future split between draft create and explicit confirm without losing event semantics.

Allowed simplified implementation:

```text
reservation.confirmed
```

Only if implementation later documents that Create Reservation V1 atomically creates and confirms in one business action. Even in that case, AuditLog and StateTransitionLog must still make creation and confirmation evidence clear.

Event requirements:

- `tenantId`
- `storeId`
- `eventType`
- `targetType = reservation`
- `targetId = reservationId`
- `actorId`
- `actorType`
- `source`
- `beforeState`
- `afterState`
- `idempotencyKey`
- `metadata`
- `occurredAt`

## 12. Idempotency Boundary

Action:

```text
create_reservation
```

Required key:

```text
idempotencyKey
```

Request hash must include the normalized command intent:

- tenantId
- storeId
- partySize
- reservedStartAt UTC
- reservedEndAt UTC
- customerId
- customerName
- customerNickname
- phoneE164
- note
- reservationCode if supplied
- source

Behavior:

| Existing status | Same hash behavior | Different hash behavior |
| --- | --- | --- |
| `completed` | Replay previous success result with `replayed=true`. | `IDEMPOTENCY_CONFLICT`. |
| `started` | Return retry-later/conflict application error. | `IDEMPOTENCY_CONFLICT`. |
| `failed` | Return `IDEMPOTENCY_FAILED_REQUIRES_NEW_KEY`. | `IDEMPOTENCY_CONFLICT`. |
| missing | Start record and execute once. | Not applicable. |

Missing key:

```text
MISSING_IDEMPOTENCY_KEY
```

Do not create duplicate Reservation for a retried same command.

## 13. Failure Cases

| Scenario | Application error code | Write audit | Mark idempotency failed | Retry allowed |
| --- | --- | --- | --- | --- |
| Store not found | `STORE_NOT_FOUND` | Yes, target null | Yes, if record started | Yes with corrected store |
| Store scope mismatch | `STORE_SCOPE_MISMATCH` | Yes, target null | Yes, if record started | Yes with corrected scope/key |
| Missing idempotency key | `MISSING_IDEMPOTENCY_KEY` | No or best-effort pre-command audit | No record exists | Yes with key |
| Invalid party size | `INVALID_PARTY_SIZE` | Yes, target null | Yes, if record started | Yes with corrected request/new key if failed |
| Invalid time range | `INVALID_TIME_RANGE` | Yes, target null | Yes, if record started | Yes with corrected request/new key if failed |
| `reservedEndAt <= reservedStartAt` | `INVALID_TIME_RANGE` | Yes, target null | Yes, if record started | Yes with corrected request/new key if failed |
| `reservedStartAt` in the past | `RESERVATION_START_IN_PAST` | Yes, target null | Yes, if record started | Yes with future time/new key if failed |
| Customer not found | `CUSTOMER_NOT_FOUND` | Yes, target null | Yes, if record started | Yes with corrected customer/new key if failed |
| Invalid phone E.164 | `INVALID_PHONE_E164` | Yes, target null | Yes, if record started | Yes with corrected phone/new key if failed |
| Duplicate active reservation | `DUPLICATE_ACTIVE_RESERVATION` | Yes, target null or existing target | Yes, if record started | No for same request; use different time or cancel existing |
| Capacity unavailable | `RESERVATION_CAPACITY_UNAVAILABLE` | Yes, target null | Yes, if record started | Yes after capacity/time changes; otherwise no |
| Store policy missing | `STORE_POLICY_MISSING` | Yes, target null | Yes, if record started | Yes after policy fix/new key |
| Idempotency conflict | `IDEMPOTENCY_CONFLICT` | Recommended | No state change to existing record | No with same key; use new key |
| Idempotency in progress | `IDEMPOTENCY_IN_PROGRESS` | No or recommended | No | Yes later with same key |
| Failed key requires new key | `IDEMPOTENCY_FAILED_REQUIRES_NEW_KEY` | No or recommended | Existing failed record unchanged | Yes with new key |
| BusinessEvent write failure | `BUSINESS_EVENT_WRITE_FAILED` | Yes if audit still possible | Yes, if record started | Yes with new key after fix |
| Audit write failure | `AUDIT_WRITE_FAILED` | No, audit failed | Yes, if record started | Yes with new key after fix |
| State transition failure | `STATE_TRANSITION_WRITE_FAILED` | Yes if audit still possible | Yes, if record started | Yes with new key after fix |
| Persistence failure | `PERSISTENCE_ERROR` | Yes if audit still possible | Yes, if record started | Yes with new key after fix |

Failure transaction guidance:

- Reservation, BusinessEvent, StateTransitionLog, AuditLog, and Idempotency completion should commit atomically on success.
- If a business mutation is rolled back, no partial Reservation should remain.
- Failed idempotency persistence is best-effort within V1 behavior, matching current vertical-slice patterns.

## 14. Test Contract

This round does not write tests. A later implementation round should cover the following.

Success:

- Create reservation with existing Customer.
- Create reservation with no-phone temporary Customer.
- Create reservation with phone Customer.
- Create reservation stores UTC instants.
- Create reservation derives Store-local `businessDate`.
- Create reservation status is `confirmed`.
- Create reservation sets `holdUntilAt` using Store reservation hold policy.
- Writes `reservation.created` and `reservation.confirmed`, or documented single event.
- Writes StateTransitionLog.
- Writes AuditLog.
- Completes idempotency.

Idempotency:

- Completed replay returns previous result with `replayed=true`.
- In-progress request returns retry-later/conflict error.
- Failed key requires new key.
- Same key with different request hash returns conflict.
- Same command retried with completed key does not create duplicate Reservation.

Failure:

- Invalid party size.
- Invalid time range.
- Past start time.
- Duplicate active reservation.
- Capacity unavailable.
- Invalid phone.
- Customer not found.
- Store scope mismatch.
- Store policy missing.
- Audit write failure.
- Repository/persistence failure.

Boundary:

- No QueueTicket created.
- No Seating created.
- No TableLock created.
- No CheckIn created.
- No No-show transition.
- No Reservation API implemented in this application contract round.
- No UI implemented.
- No migration changed.
- No production database touched.
